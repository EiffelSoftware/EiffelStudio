21 USING FORMATS

           Help reading this manual

The System, Class and Feature tools that we have seen until now all showed the corresponding text: Ace text., class text, feature text. It is also possible to display all kinds of other information about the corresponding development objects (systems, classes, features), such as the classes of the system or the ancestors of a class. This format mechanism makes the drag-and-drop browsing mechanism even more exciting.

System Tool formats

Let's start with the formats of the System Tool. If you don't see the System Tool on your screen, go to the Project Tool and click the system hole (second from the left in the top row of icons). This brings the System Tool back up:

The second row of icons in the System Tool lists all the available formats. As usual, you can go over them with the mouse, without clicking, to see the tooltips that give you a preview of what they mean.

The format selected by default is the Text format , the first one from the left in the System Tool and other tools. Here it displays the Ace text, which you used a while ago to drag-and-drop the root class, TESTROOT . For the moment nothing else is clickable.

The second format from the left, , is the Clusters format. Click it now. It gives you another view of the system, showing all its classes, cluster by cluster:

    Reminder: clusters in the Eiffel method

    An Eiffel system is divided into clusters. Each cluster is a group of classes, each collected in a directory.

    The list of clusters and their associated directories is given by the Ace file (which does not need, however, to list the clusters of precompiled libraries).

Here we see 17 clusters, all of which but the root cluster are part of precompiled EiffelBase (that's why the Ace file only listed the root cluster). Altogether there are 157 classes. Note that the actual figures that you get may be somewhat different, depending on your version of EiffelBase. Within each cluster, classes are listed alphabetically. The root cluster always appears first, the precompiled clusters last.

As usual, all the class names are clickable. So if you are curious about some class, say ARRAY , just control-right-click it to get a Class Tool that will show its text. (You can also drag-and-drop it to a Class Hole.)

Coming back to the Class Tool, let us discover the other formats. The third one from the left, , is Classes. Click it and then click Continue in the panel that comes up (" This format requires exploring the entire system and may take a long time … Continue/Cancel "). You get -- after a short delay -- an alphabetical list of classes, each with its cluster:

Next comes the Hierarchy format . In Lace you can specify that a cluster is a subcluster of another, through the following syntax:

where my_subcluster_directory can include the symbol $ to refere to the directory of the parent cluster some_parent_cluster -- greatly simplifying Ace files. click Hierarchy to see the resulting cluster hierarchy for our example -- consisting mostly of the cluster hierarchy for EiffelBase. This hierarchy is shown here under Unix (the exact hierarchy may vary with the version of the environment):

The next format, Statistics, produces some information about the system. Try it now:

Again the exact numbers may vary when you run the Tour.

The next format, Modified Classes , tells you what classes have been modified since the last compilation. You can try it if you like, although it won't show anything interesting since we have not modified any class yet.

Try now the last format, Indexing . It warns you again that it may take a longer time than the others (confirm anyway by clicking Continue ); when it comes back after a few seconds, it shows the key information from the indexing clauses of all classes in the system:

The optional Indexing clause comes at the beginning of an Eiffel class and enables the class author to associate indexing information with the class. The Indexing format retains only some entries of the Indexing clause, notably description . This is a great way to look for classes possessing certain properties. (To search for a certain string in the text of a tool, be it the System Tool or another, you can use the Search button, , which appears on the right of the top row of buttons.)

A common property of all the formats of the System Tool is that all the class names are clickable; so you can immediately see the properties of any class that catches your attention.

Class Tool formats

Let us now look at the formats of the Class Tool. Since we have had enough of the little classes of our example -- TESTROOT , PARENT and HEIR -- let us use an EiffelBase class, TABLE . By now you know several ways of targeting a Class Tool to that class; for example, if you still have a Class Tool up, just click into its class name field, and type table followed by Enter. (If you don't have a Class Tool up any more, you can get an empty one by just left-clicking on the class hole of the Project Tool, used here as a "buttonhole". Then retarget the Class Tool to TABLE by typing its name as just explained.)

The second row of icons lists all the Class Tool formats. As with the System Tool, the default format -- the only one used so far -- is Text , which displays the text of a class as it has been entered by the class author.

The second format from the left is Clickable. Try it now:

The Clickable format shows a class in a reformated ("pretty-printed") form: keywords in boldface and in blue, identifiers in italics, comments in roman and in red, and so on. (You can change the color and font conventions by bringing up the Preference Tool through the Windows -> Preferences menu entries in the Project Tool, although there is no need to do this right now.) Scroll through the tool to see some of the text of class TABLE pretty-printed.

The Clickable format is called that way because it makes many more elements clickable than the default Text format.. Bring TABLE back to the Text format temporarily and see where you can right-click (in each case please left-click immediately afterwards to cancel the incipient drag-and-drop: you can right-click the class name TABLE after deferred class , its parent BAG , feature names such as item at the place of their declaration, or such as put in the rename clause. But you cannot click the components of a routine body, such as valid_key in the body of item . But in the Clickable format every semantically meaningful component is clickable. For example, after clicking to switch back to Clickable format, you can control-right-click valid_key (the actual call, in italics, not the label). This brings up a Feature Tool targeted to that feature. Since we don't need that tool for the moment, just click its Quit button to dismiss it.

Ignore the Flat button and click Short (fourth format button from the left) to get the short form of class TABLE :

The short form, an important concept in the Eiffel method, provides the official interface that a class offers to its client. It is obtained from the class text by removing all implementation information -- routine bodies, all secret features -- but keeping the routine headers, the header comments and the assertions. This is the standard interface through which you will use a class: informative enough to tell you what the class "does", but freed from all details that matter only to the class's authors, not to its users.

The Flat form provides the reconstructed form of a class, with all inherited features; and the Flat-short form provides the short of the flat, that is to say the complete interface. We ignore them for the moment. (Do look at them if you like; note that they require exploring the entire ancestry of a class, and so take longer to produce for a class such as TABLE which, as we are going to see, is has a rich ancestry.)

Click now the Ancestors format ; this gives the ancestor structure that leads to the class:

This shows that TABLE inherits from BAG and so on. The inheritance structure stops at ANY , ancestor of all developer-written classes. But look at the ancestors of ANY itself by retargeting the tool: right-clik on ANY to pick that class, and right-click again (with or without moving the mouse in-between) to drop and hence retarget. Note that when you retarget a tool it stays in the format that it had before; so here the Class Tool stays in Ancestors format and hence shows the ancestors of ANY :

In the case of ANY (only for that class and for PLATFORM ) the displayed inheritance structure includes the two top-level classes: PLATFORM and GENERAL . GENERAL is where all the general-purpose features, useful to all classes, are defined: clone , copy , equal and the like. For classes other than ANY , the Ancestor format stops at ANY , not showing PLATFORM and GENERAL which are always there and would clutter the display.

Now assume you want to take a peek at GENERAL . You now know how to do this: drag-and-drop GENERAL to retarget the current tool to that class; by definition the result is not fascinating because the tool is in Ancestors format and GENERAL has no ancestors:

Take a look at GENERAL now if you wish, or leave this for some future session. Now come back to Text format by clicking the corresponding button , and retarget the tool to TABLE ; the easiest way to do this (although by now you know several others) is to click twice the Previous history button , which takes us back to the previous targets: ANY , then TABLE .

The next format you should try is Descendants :

Class TABLE indeed has a large and prosperous progeny, as explained in detail in the book about the libraries. As always, all the names shown are clickable, so that if a class or other development catches your attention in a tool under any format you can in a snap find information about it by targeting a tool to it (the current one, or a new one). This is a good opportunity to mention the following practical advice:

    Tip: Two can be better than one

    You may find it convenient, when doing extensive browsing, to keep two class tools open and drag-and-drop freely from one to the other.

    This enables you to keep focused on a class while exporing properties of others to which it is related.

The remaining Class Tool formats are Attributes , Routines , Deferred , Once , Externals , Exported . They enable you to explore various subsets of the features of a class. We won't try them all right now, but please click now on Routines , the second one from the left on the right row of format buttons. The tool shows information about the routines of class TABLE :

The routines of the class are all listed, sorted according to their originating class: some are defined in TABLE itself -- they appear in part at the bottom of the above figure -- but others come from proper ancestors CONTAINER , COLLECTION , BAG .

Note the first of the features defined in TABLE itself (the last category on the figure): it shows up as _infix_@ . (If your display doesn't show it, scroll down to it.) In the class text, the feature appears under the name infix "@" ; this is the Eiffel mechanism for features used in infix form, such as infix "+" on integers. (We will look at the feature declaration in a few moments.) Here it means that a client can access the entry of key k in a table t through the notation t @ k as an alternative to t.item (k) . In list of features such as above, the conventional representation is infix_x where x is the symbol for the infix operator.

Once again everything -- class names, feature names -- is clickable in the above format, helping you explore the properties of all your development objects.

Feature tool formats

To look up infix "@" you could set the TABLE Class Tool to Text or Clickable format, but instead let us get it in a Feature Tool, both because this is directly relevant and because we will use the opportunity to discover the formats of Feature Tools. To bring up the tool, control-right-click _infix_@ at the place where this feature name appears. You see the feature declaration:

The comment, added by EiffelStudio, indicates where the feature's version comes from. Here infix "@" is declared in the class TABLE itself, but in the general case the applicable version of any feature in any class can come from any ancestor.

Note the use of Eiffel's synonym mechanism: two names are given to the same features, an identifier name item and an infix name infix "@" . The effect is the same as if we had two declarations, identical except for the feature name.

A Feature Tool has two text fields at the top: one for the feature name and one for the class name. It also has two holes at the top left: the first one represents the feature, the second one its class, here TABLE . Both holes are full, so you can drag-and-drop from them if you need to.

Let us retarget the tool to another feature, extend ; you can do this by typing the name in the first field, but instead please go to the Class Tool that is targeted to TABLE in Routines format (you should still have it up, otherwise bring it back, you know how to do this!), find extend , which appears at the beginning of the routines coming from class BAG , and start drag-and-drop it to the Feature Tool. In principle you should drop it into the feature hole at the top left , but in fact any place in the window's main area -- the white-background space where the text of infix "@" now appears -- is fine, so please drop there. The tool retargets itself to extend from TABLE :

The notation ensure then , as you may remember, is used to change a postcondition -- by strengthening it -- in a redeclaration of an inherited feature. The counterpart for a precondition -- weakening, not used here -- is require else .

You can also retarget a Feature Tool using a class rather than a feature, for example to see the version of a feature for a descendant of the current class. To see this, go again to the Class Tool targeted to TABLE and put it in Descendants format:as was done before, with the display starting like this:


    The last line shown is for LIST , one of the grand-grandchildren of TABLE . Now pick the name of LIST and bring it to the Feature Tool. In principle you should drop it into the class hole , second from left, but as before you can actually drop it anywhere in the window's main area. Do this now. The result is to retarget the Feature Tool to the version of extend as defined in LIST .

As you may have noted, the feature is in fact the same as extend from TABLE that we saw before: both versions come, deferred, from the ancestor BAG . Descendants of LIST other than TABLE , and descendants of TABLE , provide various effectings (implementations) of that feature.

This mechanism is what you will use when you know about a feature f from a class A , available in a Feature Tool, and want to know what the version of f is for a class B : just drag-and-drop A to the tool. You can drop either a feature or a class into a Feature Tool.

The other formats of the Feature Tool will help us discover further what happens to a feature throughout its various incarnations in the inheritance structure.

First try Flat , which gives the reconstructed version of the feature with all inherited assertions combined:

The next format yields the Callers of a feature: all the places in the system where the feature is called. It takes a while to produce, you may want to skip it for the moment. If you do click it, you get this:

See how this gives the callers not just from clients of the current class, LIST but also of its descendants. In fact there is no call to extend from a direct client of LIST , that is to say no call of the form my_list . extend (some_value) for my_list of type LIST [SOME_TYPE] ; but the display shows such calls for my_list having a type based on a descendant of LIST . As you can see, for example:

  • Procedure merge from class PART_SORTED_LIST calls extend from that same class (here acting as its own client.
  • Procedures linear_representation from ARRAY , extend from ARRAYED_CIRCULAR , copy as redefined for ARRAYED_LIST , and so on, call extend from ARRAYED_LIST .

As usual everything is clickable. The Callers format is particularly precious for debugging. It enables you to see all the places in an entire system from where a routine -- perhaps one that's causing you some trouble -- gets called.

You may have noted that the title bar of the windows shows the signature of the routine as extend (v: GENERIC #1) . This is indicates that extend takes one argument, whose type is the first (and only) generic parameter of the class, called G within that class.

The next format is Implementers , which gives all the redeclared versions of a feature. Try it:

This shows all the classes where extend from LIST -- which, as you can see, originally comes from COLLECTION -- has a different, redeclared version. This feature indeed undergoes many adventures in its journey through inheritance. Now see the same information for another feature by replacing the feature name, in the left field at the top, by valid_key , and the class name, in the right field, by the name of our old friend TABLE , then hitting Enter:

Note how in this case the feature is not just redefined by also renamed; its name in HASH_TABLE , a descendant of TABLE , is valid_key . Renaming changes the name of a feature; redeclaration keeps the name but changes other properties, typically the implementation.

Retarget the feature tool to its previous target, extend from LIST , by clicking the"history back" button, and try the next format, Ancestors version :

This gives the entire history of the feature: its versions in every ancestor (not just, as with the Implementers format, those in which the feature is redeclared). Typically this format will be used with drag-and-drop: you pick an ancestor version to see what it looks like. Here there is only one " History branch ", but in multiple inheritance cases that involve joining features from several ancestors there would be several.

You can similarly try the next format, Descendants version.

The next-to-last format, Homonyms , takes longer to produce and warns you accordingly. If you decide to go ahead anyway, you will get the list of all features called extend in the entire system:

These features, unlike those shown in the last three formats, are not necessarily related to extend from LIST : they simply have the same name. (This explains why the format takes long to produce: EiffelStudio cannot rely on structural information, but must explore the entire system representation.) The large number of features called extend in many different classes are a result of the Eiffel method's rules on systematic feature naming rules for libraries, described in Reusable Software: The Base Object-Oriented Component Libraries ; extend is one of the standard names, used in container classes for the basic feature to add an element to a container.

If you have had some experience with object-oriented software construction, you will by now have realized the importance of the facilities seen so far. In particular, given the power and versatility of polymorphism and feature redeclaration (redefinition and effecting), it is essential to be able to find out quickly the answer to such questions as:

  • What does feature f become in class C ?
  • What are the descendants of C ?
  • From what class does the version of f in C come?
  • What was the original version of f ?

The techniques that we have seen already make it possible to answer such questions in the most common cases. The following chapters show many other possibilities.
