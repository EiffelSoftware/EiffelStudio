25 RUNNING A SYSTEM

           Help reading this manual

So far we have compiled our example system and browsed through it and through the libraries, but not executed anything. Let us now see how to execute a system

Cleaning up

Since you were treacherously led to introduce a validity error in the previous section, you must first make sure it is removed. The simplest way is to remove the argument to first_message in the last line of procedure display in class INVALID so that the line just reads:

    p.first_message

(Alternatively, you can change the declaration of invalid in TESTROOT back to its original comment form.) Save and melt again.

Running a system

To run the system click the Run button on the right border of the Project Tool. As you will have noted from looking at the class texts, all that the example system does is to print a few messages:

Where does the output appear? On Windows, as soon as your system produces its first output (if any), a console window appears, as shown above. All standard output will go there; this is also where your system's users will enter input if the system reads from standard input. At the end of execution, EiffelStudio outputs Press Return to finish the execution … The reason for this convention is easy to understand: you don't want the console to go away immediately at the end of the execution (after flashing the output so quickly that you won't have had the time to see it!); but you don't necessarily want the console to stay up forever, especially if you start several executions, each sprouting its own console. If you don't do anything, the console will remain, so that you can for example compare the output of several executions. To get rid of a console, just do as the message suggests: hit Return in the console. Do this now.

On Unix and VMS the matter is simpler: you have started EiffelStudio from a certain window, by typing the command ebench ; that window is where the standard output will go, and where the execution will read from standard input. You can also redirect the standard input and output using the standard operating system mechanisms.

On any platform, you may re-execute the system as many times as you wish by clicking on Run again.

Arguments

Our example system is very simple and has no need for execution arguments. In more advanced cases you may want to pass values to the execution, such as a numeric parameter or a file name, so that you can have different executions without changing and recompiling the software.

In the Eiffel text, you can access such run-time arguments through the Kernel Library class ARGUMENTS . There is another technique -- using the arguments to the root creation procedure -- but using ARGUMENTS is the most general way. Any class of your system can inherit from ARGUMENTS and use queries argument_count to know the number of arguments passed to the execution, and argument ( i ) , for i between 1 and argument_count to access the i -th element. Class ARGUMENTS has more features; since you have EiffelStudio up, you can check the details if you wish (use the short form).

To specify execution arguments from within EiffelStudio, right-click the Run button (rather than left-click, which simply starts execution). The following panel comes up:

If this system required arguments, you would enter them here, separated by spaces. Clicking Run will start an execution with these arguments; clicking OK will simply record the arguments. Later executions will keep the same arguments even if the above panel is not up any more. To change them, right-click the Run button again to bring the panel back. (Using right-click on a command icon to set command parameters, where left-clicking on the icon simply executes the command, is a general convention also applied by EiffelStudio in several other contexts.)

Here there would be no use for arguments, so you can just dismiss this panel by clicking Cancel .
